## Purpose

The Ontology Proposals Service receives, stores, and implements ontology change proposals generated by the OSS (Ontology Suggestion System). It provides a REST API for proposal management and executes approved proposals by updating Neo4j.

## Key Responsibilities

1. **Proposal Storage**
   - Receives proposals from OSS via REST API
   - Stores in PostgreSQL with full metadata
   - Provides query and filtering capabilities

2. **Proposal Implementation**
   - Executes approved proposals on Neo4j
   - Handles ISO code corrections
   - Fixes missing properties
   - Updates proposal status (IMPLEMENTED/FAILED)

3. **Data Quality Fixes**
   - ISO 3166-1 alpha-2 code corrections for countries
   - Missing property population
   - Property standardization

4. **Status Management**
   - PENDING вҶ’ ACCEPTED вҶ’ IMPLEMENTED workflow
   - Error handling (FAILED status on errors)
   - Timestamp tracking for audit trail

## Dependencies

- **PostgreSQL:** Proposal storage
- **Neo4j:** Target for implementations
- **OSS Service:** Source of proposals

## Configuration

Key environment variables:
- `POSTGRES_HOST` - Database host (default: postgres)
- `POSTGRES_PORT` - Database port (default: 5432)
- `POSTGRES_USER` - Database user
- `POSTGRES_PASSWORD` - Database password
- `POSTGRES_DB` - Database name
- `NEO4J_URI` - Neo4j connection for implementations

## APIs

**POST /api/v1/ontology/proposals** - Create proposal
**GET /api/v1/ontology/proposals** - List proposals (with filters)
**GET /api/v1/ontology/proposals/{id}** - Get specific proposal
**POST /api/v1/ontology/proposals/{id}/implement** - Execute proposal
**GET /health** - Health check

See: [Ontology Proposals API Documentation](../api/ontology-proposals-api.md)

## Database Schema

**Table:** `public.ontology_proposals`

**Key Columns:**
- `id` - Primary key (auto-increment)
- `proposal_id` - Unique identifier (from OSS)
- `status` - PENDING | ACCEPTED | REJECTED | IMPLEMENTED | FAILED
- `severity` - CRITICAL | HIGH | MEDIUM | LOW
- `change_type` - Type of change
- `evidence` - JSONB with proof
- `impact_analysis` - JSONB with assessment
- `implementation_result` - JSONB with results
- `implemented_at` - Timestamp

**Indexes:** 8 indexes for efficient filtering and queries

## Implementation Logic

### ISO Code Corrections

**Detection:** Title/description contains "ISO country code"
**Action:**
1. Extract country name from evidence
2. Lookup ISO code via `iso_codes.get_iso_code()`
3. Update entity_id in Neo4j
4. Record changes in implementation_result

**Example:**
```
"United States" вҶ’ entity_id: "US"
"North Korea" вҶ’ entity_id: "KP"
```

### Missing Properties

**Detection:** Title contains "missing required properties"
**Action:**
1. Identify entities with NULL properties
2. Set default/derived values
3. Update Neo4j nodes
4. Record count in implementation_result

### Success Tracking

```json
{
  "success": true,
  "entities_updated": 15,
  "properties_fixed": ["entity_id"],
  "fix_type": "iso_code_correction",
  "details": {
    "United States": "US",
    "Australia": "AU"
  }
}
```

## ISO Code Utilities

**Location:** `app/utils/iso_codes.py`

**Features:**
- 96 country mappings (ISO 3166-1 alpha-2)
- Fuzzy name matching
- Format validation

**Functions:**
- `get_iso_code(name)` - Returns code or None
- `validate_iso_code(code)` - Returns True/False
